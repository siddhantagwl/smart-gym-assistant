### Environment

`.env` is gitignored and holds the two `EXPO_PUBLIC_*` vars consumed by the sync services:

- `EXPO_PUBLIC_GSHEETS_WEBHOOK_URL` ‚Äî Apps Script Web App URL
- `EXPO_PUBLIC_GSHEETS_SECRET` ‚Äî shared secret sent in the POST body

Both must be present for any sync (`services/exerciseLibrarySync.ts`, `services/googleSheetsSync.ts`) to run. `EXPO_PUBLIC_` is required because Expo only exposes vars with that prefix to runtime code.

Native `/ios` and `/android` directories are gitignored ‚Äî they are regenerated by `expo run:*` and should not be committed.

## Architecture

### Layered model

```
UI (app/, components/)  ‚îÄ‚îÄreads‚îÄ‚îÄ‚ñ∂  SQLite (db/)  ‚óÄ‚îÄ‚îÄsync‚îÄ‚îÄ‚ñ∂  Google Sheets (services/)
                                       ‚ñ≤
                                       ‚îî‚îÄ‚îÄ domain/ (in-memory types)
```

Two principles drive almost every design choice ‚Äî violating either is usually a bug:

1. **SQLite is the runtime source of truth.** Screens read directly from `db/*.ts` helpers (`db.getAllSync`, etc.). The UI never awaits the network during a workout.
2. **Sync is opt-in and asynchronous.** It runs on app boot (library only, if empty) or via explicit Settings actions. Network failure must never break logging.

### Bootstrap sequence

`app/_layout.tsx` blocks rendering until DB is ready:

1. `initDb()` from [db/schema.ts](db/schema.ts) creates the three tables (`sessions`, `exercises`, `exercise_library`) and calls `seedExerciseLibrary()` from the hardcoded fallback list.
2. `ensureExerciseLibrary()` from [services/exerciseLibrarySync.ts](services/exerciseLibrarySync.ts) checks if `exercise_library` is empty; if so it tries the remote sync, and on failure falls back to the local seed.
3. `setDbReady(true)` ‚Äî only then does the Stack mount.

Returning `null` while not ready is intentional. Don't render fallback UI here; the goal is to guarantee no screen reads from an uninitialised DB.

### Routing

Uses `expo-router` with file-based routing and `typedRoutes` experiment on (`app.json`).

- `app/(tabs)/` ‚Äî the four tabs: `index` (Home), `history`, `exercises`, `settings`
- `app/session-details.tsx` ‚Äî modal presentation (configured in `app/_layout.tsx`)
- `app/manual-session.tsx`, `app/exercise-details.tsx` ‚Äî stack screens

### Domain vs. stored types

This split is deliberate and load-bearing ‚Äî preserve it when adding new fields:

- `domain/session.ts`, `domain/exercise.ts` ‚Äî in-memory shapes used by UI/business logic. Dates are `Date` objects.
- `db/sessions.ts:StoredSession`, `db/exercises.ts:StoredExercise` ‚Äî persistence shapes. Dates are ISO strings, JSON columns (`session_labels`, `tags`) are stringified.

Mapping happens at the DB-helper boundary. UI components should consume domain types; DB helpers translate.

### DB helpers

All SQL lives under `db/` ‚Äî never inline `db.runSync` in components. Conventions:

- Synchronous APIs throughout (`expo-sqlite`'s `*Sync` methods). The codebase relies on this ‚Äî don't switch to async helpers without a reason.
- Multi-statement writes (manual-session insert, sheet restore, library replace) use explicit `BEGIN; ‚Ä¶ COMMIT; / ROLLBACK;` blocks.
- "Raw" inserters (`insertSessionRaw`, `insertExerciseRaw`) exist for restore paths that bypass domain mapping. Don't use them from UI flows.
- `getRecentSessions` filters out `note = '__DISCARDED__'` ‚Äî that sentinel marks soft-deleted sessions.

### Sync semantics (read before touching)

Two distinct sync flows with different strategies:

- **Exercise library** (Sheets ‚Üí SQLite): full replace inside a transaction. Sheets is authoritative; local changes are not preserved.
- **Sessions + exercises** (SQLite ‚Üí Sheets): full snapshot POST; the Apps Script handles dedup by ID. Strict mode only inserts exercises for sessions newly inserted in the same call. Restore flow wipes local sessions/exercises and re-inserts from Sheets.

Both rely on `EXPO_PUBLIC_GSHEETS_WEBHOOK_URL` + `EXPO_PUBLIC_GSHEETS_SECRET`.

### Rest timer (non-obvious)

The rest timer in `components/ActiveSession.tsx` is **timestamp-based**, not countdown-based. The visible 90s countdown is cosmetic; actual rest is `now - restStartRef` accumulated into `accumulatedRest` and only finalized on Start Set / Finish Exercise / manual close. The previous countdown-based implementation lost data and produced 0-second rests under race conditions ‚Äî see ARCHITECTURE.md ¬ß3.2 before "fixing" anything that looks like overcomplication here.

### Path alias

`@/*` maps to repo root (`tsconfig.json`). Use it for all cross-directory imports ‚Äî e.g. `@/db/sessions`, `@/components/AddExercise`.

### Misc

- `newArchEnabled: true` and `experiments.reactCompiler: true` in `app.json` ‚Äî the React Compiler is on, so avoid manual memoization unless profiling shows a need.
- Theming uses `@/constants/theme` + `@/hooks/use-color-scheme`. Several screens (e.g. Home) hardcode a dark palette inline ‚Äî mixed pattern, follow the file you're editing.
