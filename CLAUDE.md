# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

A deeper system overview (rest-timer rationale, sync semantics, sequence diagrams) lives in [ARCHITECTURE.md](ARCHITECTURE.md). Read it before making changes that touch the rest timer, the Google Sheets sync pipeline, or the exercise-library seed.

## Commands

```bash
npm install              # install deps (run once / after pulling)
npm start                # expo start — Metro + dev menu
npm run ios              # native iOS build via expo run:ios
npm run android          # native Android build via expo run:android
npm run web              # web build (expo start --web)
npm run lint             # expo lint (eslint-config-expo flat config)
```

There is no test runner configured. Type-check via `npx tsc --noEmit` (strict mode is on).

The `reset-project` script in `package.json` references `./scripts/reset-project.js`, which does not exist — ignore it.

### Target platform

**This app is developed and tested on Android.** iOS scripts exist in `package.json` but are not the primary target. When reasoning about platform-specific behavior (background execution, notifications, permissions, OS lifecycle), default to Android specifics — Doze mode, battery optimization, foreground services, notification channels — not iOS app suspension semantics.

### Versioning

The Home screen shows `app.json` → `expo.version` at the bottom so the user can tell which build is on their device. **Bump this version on every meaningful change** before reporting work as done — don't wait to be asked. Follow semver:

- **patch** (`1.0.0` → `1.0.1`): bug fixes, copy tweaks, small UI adjustments
- **minor** (`1.0.1` → `1.1.0`): new features, new screens, schema additions
- **major** (`1.1.0` → `2.0.0`): breaking changes (DB wipe required, sync contract change, etc.)

Edit `app.json` `expo.version` as part of the same change. The user rebuilds the APK against this version, so an un-bumped version means they can't tell whether their device has the new code.

### Environment

`.env` is gitignored and holds the two `EXPO_PUBLIC_*` vars consumed by the Sheets sync:

- `EXPO_PUBLIC_GSHEETS_WEBHOOK_URL` — Apps Script Web App URL
- `EXPO_PUBLIC_GSHEETS_SECRET` — shared secret sent in the POST body

Both must be present for `services/googleSheetsSync.ts` (sessions + exercises sync, and restore) to run. `EXPO_PUBLIC_` is required because Expo only exposes vars with that prefix to runtime code.

Native `/ios` and `/android` directories are gitignored — they are regenerated by `expo run:*` and should not be committed.

## Architecture

### Layered model

```
UI (app/, components/)  ──reads──▶  SQLite (db/)  ◀──sync──▶  Google Sheets (services/)
                                       ▲
                                       └── domain/ (in-memory types)
```

Two principles drive almost every design choice — violating either is usually a bug:

1. **SQLite is the runtime source of truth.** Screens read directly from `db/*.ts` helpers (`db.getAllSync`, etc.). The UI never awaits the network during a workout.
2. **Sync is opt-in and asynchronous.** It runs on app boot (library only, if empty) or via explicit Settings actions. Network failure must never break logging.

### Bootstrap sequence

`app/_layout.tsx` blocks rendering until DB is ready:

1. `initDb()` from [db/schema.ts](db/schema.ts) creates the three tables (`sessions`, `exercises`, `exercise_library`), runs `ensureColumn()` ALTER migrations for older DBs, and calls `seedExerciseLibrary()` from the hardcoded list in `db/seedExercises.ts` (idempotent — uses `INSERT OR IGNORE`).
2. `setDbReady(true)` — only then does the Stack mount.

Returning `null` while not ready is intentional. Don't render fallback UI here; the goal is to guarantee no screen reads from an uninitialised DB.

There is **no** remote library sync on boot. The exercise library is local-only, sourced from the hardcoded seed file.

### Routing

Uses `expo-router` with file-based routing and `typedRoutes` experiment on (`app.json`).

- `app/(tabs)/` — the four tabs: `index` (Home), `history`, `exercises`, `settings`
- `app/session-details.tsx` — modal presentation (configured in `app/_layout.tsx`)
- `app/manual-session.tsx`, `app/exercise-details.tsx` — stack screens

### Domain vs. stored types

This split is deliberate and load-bearing — preserve it when adding new fields:

- `domain/session.ts`, `domain/exercise.ts` — in-memory shapes used by UI/business logic. Dates are `Date` objects.
- `db/sessions.ts:StoredSession`, `db/exercises.ts:StoredExercise` — persistence shapes. Dates are ISO strings, JSON columns (`session_labels`, `tags`) are stringified.

Mapping happens at the DB-helper boundary. UI components should consume domain types; DB helpers translate.

### DB helpers

All SQL lives under `db/` — never inline `db.runSync` in components. Conventions:

- Synchronous APIs throughout (`expo-sqlite`'s `*Sync` methods). The codebase relies on this — don't switch to async helpers without a reason.
- Multi-statement writes (manual-session insert, sheet restore, library replace) use explicit `BEGIN; … COMMIT; / ROLLBACK;` blocks.
- "Raw" inserters (`insertSessionRaw`, `insertExerciseRaw`) exist for restore paths that bypass domain mapping. Don't use them from UI flows.
- `getRecentSessions` filters out `note = '__DISCARDED__'` — that sentinel marks soft-deleted sessions.

### Sync semantics (read before touching)

Only one sync flow exists, and it covers sessions + exercises only:

- **Sessions + exercises** (SQLite → Sheets, in `services/googleSheetsSync.ts`): full snapshot POST; the Apps Script handles dedup by ID. Restore (GET) flow wipes local sessions/exercises and re-inserts from Sheets via `insertSessionRaw` / `insertExerciseRaw`.

The **exercise library is not synced**. It is seeded once from `db/seedExercises.ts` and only updated by editing that source file (or, after Pass 2, via in-app library writes — TBD). Past versions of these docs claimed a Sheets→library sync; that code never landed.

The Sheets sync requires `EXPO_PUBLIC_GSHEETS_WEBHOOK_URL` + `EXPO_PUBLIC_GSHEETS_SECRET`.

### Rest timer (non-obvious)

The rest timer in `components/ActiveSession.tsx` is **timestamp-based**, not countdown-based. The visible 90s countdown is cosmetic; actual rest is `now - restStartRef` accumulated into `accumulatedRest` and only finalized on Start Set / Finish Exercise / manual close. The previous countdown-based implementation lost data and produced 0-second rests under race conditions — see ARCHITECTURE.md §3.2 before "fixing" anything that looks like overcomplication here.

**Background behavior (Android):** when the app is backgrounded, the JS thread is suspended/throttled, so the visible countdown freezes and `setInterval` stops firing. The *saved* rest value remains correct because it's derived from wall-clock timestamps when the user taps Start Set / Finish Exercise. There is currently **no** local notification, foreground service, or `AppState` handling — the user gets no alert when rest is up if the app is not in the foreground. `expo-notifications` is not installed.

### Path alias

`@/*` maps to repo root (`tsconfig.json`). Use it for all cross-directory imports — e.g. `@/db/sessions`, `@/components/AddExercise`.

### Misc

- `newArchEnabled: true` and `experiments.reactCompiler: true` in `app.json` — the React Compiler is on, so avoid manual memoization unless profiling shows a need.
- Theming uses `@/constants/theme` + `@/hooks/use-color-scheme`. Several screens (e.g. Home) hardcode a dark palette inline — mixed pattern, follow the file you're editing.
