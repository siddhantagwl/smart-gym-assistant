# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

A deeper system overview (rest-timer rationale, sync semantics, sequence diagrams) lives in [ARCHITECTURE.md](ARCHITECTURE.md). Read it before making changes that touch the rest timer, the Google Sheets sync pipeline, or the exercise-library seed.

## Commands

```bash
npm install              # install deps (run once / after pulling)
npm start                # expo start — Metro + dev menu
npm run ios              # native iOS build via expo run:ios
npm run android          # native Android build via expo run:android (debug)
npm run android:release  # release-variant Android build (the APK the user installs)
npm run web              # web build (expo start --web)
npm run lint             # expo lint (eslint-config-expo flat config)
```

There is no test runner configured. Type-check via `npx tsc --noEmit` (strict mode is on).

The `reset-project` script in `package.json` references `./scripts/reset-project.js`, which does not exist — ignore it.

### Target platform

**This app is developed and tested on Android.** iOS scripts exist in `package.json` but are not the primary target. When reasoning about platform-specific behavior (background execution, notifications, permissions, OS lifecycle), default to Android specifics — Doze mode, battery optimization, foreground services, notification channels — not iOS app suspension semantics.

### Versioning

The Home screen shows `app.json` → `expo.version` at the bottom so the user can tell which build is on their device. **Bump this version on every meaningful change** before reporting work as done — don't wait to be asked. Follow semver:

- **patch** (`1.0.0` → `1.0.1`): bug fixes, copy tweaks, small UI adjustments
- **minor** (`1.0.1` → `1.1.0`): new features, new screens, schema additions
- **major** (`1.1.0` → `2.0.0`): breaking changes (DB wipe required, sync contract change, etc.)

Edit `app.json` `expo.version` as part of the same change. The user rebuilds the APK against this version, so an un-bumped version means they can't tell whether their device has the new code.

### Environment

`.env` is gitignored and holds these `EXPO_PUBLIC_*` vars:

- `EXPO_PUBLIC_GSHEETS_WEBHOOK_URL` — Apps Script Web App URL (required for any sync)
- `EXPO_PUBLIC_GSHEETS_SECRET` — shared secret sent in the POST body (required)
- `EXPO_PUBLIC_GSHEETS_SPREADSHEET_URL` — direct URL of the spreadsheet, optional. If set, Settings shows an "Open spreadsheet ↗" link that calls `Linking.openURL`. Independent of sync — sync works without it.

The first two must be present for `services/googleSheetsSync.ts` (sessions/exercises sync, restore, library sync) to run. `EXPO_PUBLIC_` is required because Expo only exposes vars with that prefix to runtime code, and these are inlined into the JS bundle at build time — missing values silently ship as `undefined`. `services/googleSheetsSync.ts` exports `isSyncConfigured()`; Settings renders a red "Sync not configured" banner when it's false, so a busted build is visible at a glance instead of presenting as mystery sync errors.

For local builds, `.env` at project root is read by Metro at bundle time. For EAS Build, register the same vars via `eas env:create` (EAS does not upload your local `.env`).

Native `/ios` and `/android` directories are gitignored — they are regenerated by `expo run:*` and should not be committed.

## Architecture

### Layered model

```
UI (app/, components/)  ──reads──▶  SQLite (db/)  ◀──sync──▶  Google Sheets (services/)
                                       ▲
                                       └── domain/ (in-memory types)
```

Two principles drive almost every design choice — violating either is usually a bug:

1. **SQLite is the runtime source of truth.** Screens read directly from `db/*.ts` helpers (`db.getAllSync`, etc.). The UI never awaits the network during a workout.
2. **Sync is opt-in and asynchronous.** It runs on app boot (library only, if empty) or via explicit Settings actions. Network failure must never break logging.

### Bootstrap sequence

`app/_layout.tsx` blocks rendering until DB is ready:

1. `initDb()` from [db/schema.ts](db/schema.ts) creates the three tables (`sessions`, `exercises`, `exercise_library`), runs `ensureColumn()` ALTER migrations for older DBs, and calls `seedExerciseLibrary()` from the hardcoded list in `db/seedExercises.ts` (idempotent — uses `INSERT OR IGNORE`).
2. `setDbReady(true)` — only then does the Stack mount.

Returning `null` while not ready is intentional. Don't render fallback UI here; the goal is to guarantee no screen reads from an uninitialised DB.

There is no automatic library sync on boot. `seedExerciseLibrary()` only fills the table on first install (idempotent). The user runs `syncExerciseLibrary()` manually from Settings to pull the canonical library from Google Sheets.

### Routing

Uses `expo-router` with file-based routing and `typedRoutes` experiment on (`app.json`).

- `app/(tabs)/` — the four tabs: `index` (Home), `history`, `exercises`, `settings`
- `app/session-details.tsx` — modal presentation (configured in `app/_layout.tsx`)
- `app/manual-session.tsx`, `app/exercise-details.tsx` — stack screens

### Domain vs. stored types

This split is deliberate and load-bearing — preserve it when adding new fields:

- `domain/session.ts`, `domain/exercise.ts` — in-memory shapes used by UI/business logic. Dates are `Date` objects.
- `db/sessions.ts:StoredSession`, `db/exercises.ts:StoredExercise` — persistence shapes. Dates are ISO strings, JSON columns (`session_labels`, `tags`) are stringified.

Mapping happens at the DB-helper boundary. UI components should consume domain types; DB helpers translate.

### DB helpers

All SQL lives under `db/` — never inline `db.runSync` in components. Conventions:

- Synchronous APIs throughout (`expo-sqlite`'s `*Sync` methods). The codebase relies on this — don't switch to async helpers without a reason.
- Multi-statement writes (manual-session insert, sheet restore, library replace) use explicit `BEGIN; … COMMIT; / ROLLBACK;` blocks.
- "Raw" inserters (`insertSessionRaw`, `insertExerciseRaw`) exist for restore paths that bypass domain mapping. Don't use them from UI flows.
- `getRecentSessions` filters out `note = '__DISCARDED__'` — that sentinel marks soft-deleted sessions.

### Sync semantics (read before touching)

Two sync flows, both in `services/googleSheetsSync.ts`:

- **Sessions + exercises** (SQLite → Sheets, `syncToGoogleSheets`): full snapshot POST; the Apps Script handles dedup by ID. Restore (`restoreFromGoogleSheets`, GET no-param) wipes local sessions/exercises and re-inserts from Sheets via `insertSessionRaw` / `insertExerciseRaw`. Restore does **not** touch `exercise_library`.
- **Exercise library** (Sheets → SQLite, `syncExerciseLibrary`): GET `?type=library`, full-replace inside a transaction (`replaceExerciseLibrary` in `db/exerciseLibrary.ts`). Sheet is authoritative; locally added entries are wiped on next sync. The hardcoded seed in `db/seedExercises.ts` only fills the table on first install via `INSERT OR IGNORE` — once the user runs library sync, they're effectively replacing the seed.

Both sync flows require `EXPO_PUBLIC_GSHEETS_WEBHOOK_URL` + `EXPO_PUBLIC_GSHEETS_SECRET`.

### Rest timer (non-obvious)

The rest timer in `components/ActiveSession.tsx` is **timestamp-based**, not countdown-based. `restStartRef.current = new Date()` is the single source of truth; everything else (visible counter, scheduled notification, saved rest seconds) is derived from it.

Three cooperating pieces, all keyed off `restStartRef`:

1. **Visible counter** (`restSeconds` state) — recomputed each `setInterval` tick as `duration - secondsBetween(new Date(), restStartRef.current)`, **not** decremented. So when Android suspends JS in the background and the interval pauses, returning to the foreground snaps the counter to the correct elapsed value within ~1s. Don't refactor this back to `prev - 1` — that's the broken pattern we replaced.
2. **OS-scheduled notification** — `services/notifications.ts` schedules a local notification at `+duration` seconds via `expo-notifications`. The ID is held in `restNotificationIdRef`. Fires regardless of JS thread state (app backgrounded, screen off, app killed). Two kinds: `"set"` (90s, "Rest done") and `"transition"` (120s, "Next exercise"). Cancellation is best-effort: every `stopRestTimer()` cancels the saved ID, and every `startRestTimer()` cancels the prior ID before scheduling a new one — invariant: at most one rest notification scheduled at any time.
3. **Saved rest value** — `stopRestTimer()` computes `secondsBetween(new Date(), restStartRef.current)` and accumulates into `accumulatedRest`. Independent of whether JS was running. Finalized only on Start Set / Finish Exercise / manual close. The previous countdown-based implementation lost data and produced 0-second rests under race conditions — see ARCHITECTURE.md §3.2.

**Notification setup:** `setupNotifications()` in `services/notifications.ts` is called once on app boot from `app/_layout.tsx`. It registers the "rest-timer" Android channel (importance HIGH, vibration pattern, default sound) and requests notification permission. Idempotent. The module also calls `Notifications.setNotificationHandler` at module load to allow notifications to show even when the app is foregrounded — without this, `expo-notifications` suppresses everything while the app is on screen.

**Android battery optimization caveat:** Android's Doze mode can delay scheduled notifications by 10-30+ seconds when the device is idle (screen off, in pocket — exactly the rest scenario). To get on-time delivery, the user must set the app to "Unrestricted" battery in system Settings. We don't request the `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` permission programmatically; that's a documented one-time per-device setup. The breathing-cat ring animation (`Animated.timing` on `restRingProgress`) is also JS-driven and will visually drift on long backgrounds — known cosmetic, not blocking.

### Path alias

`@/*` maps to repo root (`tsconfig.json`). Use it for all cross-directory imports — e.g. `@/db/sessions`, `@/components/AddExercise`.

### Misc

- `newArchEnabled: true` and `experiments.reactCompiler: true` in `app.json` — the React Compiler is on, so avoid manual memoization unless profiling shows a need.
- Theming uses `@/constants/theme` + `@/hooks/use-color-scheme`. Several screens (e.g. Home) hardcode a dark palette inline — mixed pattern, follow the file you're editing.
